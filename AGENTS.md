# Preisser Solutions Website Rules

Follow `CLAUDE.md` for repo workflow and session logging. This file is the concise cross-agent rule set for SEO, local search, conversion, compliance, and repo preservation.

## Business Context

Preisser Solutions is a Hays, Kansas company owned by Tyler Preisser. It helps local businesses get more calls, bookings, and customers through websites, local SEO, Google Business Profile optimization, Google Ads, review systems, social media marketing, lead tracking, and AI automation.

## Primary Website Goals

1. Generate booked calls and free Hays Visibility Audit requests.
2. Rank and convert for Hays marketing, SEO, web design, Google Ads, GBP, and AI automation searches.
3. Build trust as a legitimate Hays and Northwest Kansas entity.
4. Support AI-search readability with clear, factual, crawlable content.
5. Avoid unsupported claims and search-policy risks.
6. Preserve repo cleanliness and avoid duplicate files.

## Content Rules

- Never invent testimonials, reviews, clients, awards, certifications, rankings, revenue numbers, or case studies.
- Mark unknown details with TODO placeholders.
- Do not keyword-stuff.
- Write for business owners, not marketers.
- Use direct, credible, local language.
- Every page should answer the searcher intent quickly.
- Avoid generic marketing fluff.

## SEO Rules

Every major page should have a unique title, unique description, one H1, canonical URL, useful internal links, visible HTML copy, relevant FAQs, and schema only when it matches visible page content.

## Conversion Rules

Primary CTA: Get a Free Hays Visibility Audit.

Secondary CTA: Call Preisser Solutions.

Avoid vague primary CTAs like Learn More.

## Local SEO Rules

- Use the real business name: Preisser Solutions.
- Do not create fake office or address claims.
- Do not create fake reviews, fake ratings, fake sponsorships, or fake press.
- Use Hays, Ellis County, and Northwest Kansas naturally where relevant.
- Keep NAP consistent wherever it appears.
- If address, GBP link, review link, or booking URL is missing, use a clear TODO instead of fabricating.

## Repo Preservation Rules

- Improve existing files before creating new files.
- Search for existing equivalents before adding pages, components, docs, schema helpers, metadata helpers, forms, CTAs, or deployment config.
- Do not create duplicate routes, docs, components, schema systems, SEO metadata systems, forms, CTAs, or deployment configs.
- Do not leave scratch, backup, copy, temp, or v2 files.
- Do not delete files unless they are proven unused and the reason is documented.
- Every new file must be routed, imported, linked, referenced, or documented.

## Technical Checks

Before final response, run available checks: typecheck, build, lint if non-interactive, schema JSON parse checks, and link/sitemap checks where practical. Report failures exactly.
